# Latent Separatrix Data Protocol

This file is a guardrail for future runs. Do not mix the two data regimes.

## Regime A: ProcessBench Full / Single-Solution Data

Canonical remote paths:

```bash
/gz-data/research/demo/data/features/full_gsm8k.npz
/gz-data/research/demo/data/features/full_math.npz
/gz-data/research/demo/data/features/full_omnimath.npz
```

Typical local relative paths on the GPU box:

```bash
data/features/full_gsm8k.npz
data/features/full_math.npz
data/features/full_omnimath.npz
```

Expected fields:

```text
stepvec or sv_vec_<mode>
gold_error_step or labels
problem_ids
steps_text
step_token_ranges
sv_layers or layers_used
```

Correct use:

```bash
python latent_separatrix_audit.py data/features/full_gsm8k.npz \
  --output_dir outputs/latent_separatrix_full_gsm8k \
  --tag full_gsm8k
```

What this regime can support:

- ProcessBench first-error localization.
- Cross-problem response diagnosis.
- Within-chain first-error ranking.

What this regime cannot support:

- Same-problem response AUROC.
- Same-problem paired correct-vs-incorrect response claims.

Expected symptom if used for same-problem evaluation:

```text
eligible_groups=0/N
undefined (eligible_groups=0/N; single-class groups=N)
```

This is not a missing-output bug. It means the selected data file has no usable
same-problem positive/negative pairs.

## Regime B: Same-Problem Multi-Sample Data

Generated by:

```bash
python 10_sample_and_extract.py ...
```

Canonical filename pattern:

```bash
data/<tag>_multisample_sv.npz
```

Common examples used by older scripts:

```bash
data/gsm8k_multisample_sv.npz
data/features/sampled_v2_5shot_features.npz
```

Expected fields:

```text
problem_ids
sample_idx
is_correct
is_correct_strict
format_ok
responses
steps_text
sv_vec_step_exp or another sv_vec_<mode>
```

Important: this regime usually does not have `gold_error_step`.

Correct use:

- Use same-problem response-level audits.
- Headline metric should be same-problem paired AUROC:

```text
P(score(incorrect sample) > score(correct sample) | same problem)
```

Do not run `latent_separatrix_audit.py` on this file as if it had ProcessBench
first-error labels. A separate response-level weak-label script is required,
for example:

```bash
python latent_multisample_response_audit.py data/gsm8k_multisample_sv.npz \
  --label is_correct_strict \
  --output_dir outputs/latent_multisample_gsm8k \
  --tag gsm8k_multisample
```

If that script does not exist yet, implement it before claiming same-problem
response results.

## Quick Decision Rule

Use this before every run:

```text
Need first-error step localization?       -> use full_*.npz + latent_separatrix_audit.py
Need same-problem response diagnosis?     -> use *_multisample_sv.npz + multisample response audit
Need within-problem paired AUROC?         -> never use full_*.npz
Need ProcessBench gold_error_step labels? -> never assume *_multisample_sv.npz has them
```

## Why This Matters

`full_*.npz` and `*_multisample_sv.npz` answer different questions.

- `full_*.npz`: "Can we localize the first annotated process error?"
- `*_multisample_sv.npz`: "For the same problem, can we rank incorrect sampled
  responses above correct sampled responses?"

Mixing them creates misleading empty within-problem tables or position-confounded
step results.
